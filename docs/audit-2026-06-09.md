## 2. Security Findings

### [CRITICAL] C-01 — `verifyPrice` has no staleness/nonce check; stale signed prices drain collateral via force-execute, borrow, and liquidate

- **Contract:** OracleVerifier.sol (+ PythOracleVerifier.sol)
- **Function:** `verifyPrice()`
- **Line:** 122-136

**Description:** The push path `updatePrice` enforces staleness (line 70), but the inline-proof path `verifyPrice` only checks `price != 0` and signature validity — **no staleness, no expiry, no sequence number**. Any price the signer has *ever* signed for an asset is a valid proof forever. The consumers all pass **caller-supplied** `priceData`:
- `OwnMarket.forceExecuteOrder` (OwnMarket.sol:202, 207) — redeemer supplies asset + collateral proofs.
- `BorrowManager.borrow / liquidate / absorbBadDebt` (BorrowManager.sol:199, 306, 407) — borrower/liquidator supplies the price.

`PythOracleVerifier.verifyPrice` has the same class of gap: it trusts the caller's `[minPublishTime, maxPublishTime]` window and never enforces `maxPriceAge` on the inline path.

**Impact:** Direct theft of LP collateral via multiple vectors:
- **Force-redeem:** pick highest-ever asset price + lowest-ever collateral price → `grossCollateral = remaining·assetPrice/collateralPrice` releases far more vault collateral than the redemption is worth.
- **Borrow:** supply a stale *high* eToken price → `collateralValueUSD` inflated → borrow stablecoins against overvalued collateral, walk away underwater. LPs eat the shortfall.
- **Liquidate:** supply a stale *low* price → make a healthy position look liquidatable → seize collateral + bonus.

Preconditions are minimal (hold a redeem order and wait `claimThreshold`; or just call `borrow`). Affects every asset regardless of oracle type.

**Proof of Concept:** (1) Signer publishes eTSLA @ $500 at T; signature S is on-chain/in logs. (2) Price falls to $300. (3) Attacker `forceExecuteOrder(orderId, vault, encode($500,T,S), encode(lowEthPrice,...))` → `verifyPrice` returns $500, timestamp ignored → ~40% over-release.

**Recommendation:** In `verifyPrice`, enforce per-asset staleness: `if (block.timestamp - timestamp > maxStaleness) revert StalePrice()`; add a per-asset monotonic sequence/nonce to the signed digest to invalidate superseded prices. In `PythOracleVerifier`, clamp `maxPublishTime` to `block.timestamp` and require `block.timestamp - publishTime <= maxPriceAge`. Have `OwnMarket`/`BorrowManager` assert the returned timestamp's freshness.

### [HIGH] H-01 — `forceExecuteOrder` accepts a halted (wind-down) vault as collateral source

- **Contract:** OwnMarket.sol
- **Function:** `forceExecuteOrder()`
- **Line:** 189, 214 (sink: OwnVault.sol:537)

**Description:** The only check on the caller-named `vault` is `isRegisteredVault` (line 189). A `Halted` vault has had its collateral excluded from the global pool (`onVaultHalted`, VaultManager.sol:256) and is winding down to its LPs, yet `releaseCollateral` (OwnVault.sol:537, `onlyMarket`, no status check) will still pay out from it, while `closeExposure` reduces the *global* book.

**Impact:** A redeemer cherry-picks the wind-down vault and drains collateral reserved for that vault's instant LP withdrawals; loss/dilution mutualized onto the halted vault's LPs beyond design intent.

**Recommendation:** In `forceExecuteOrder` require `IOwnVault(vault).vaultStatus() == VaultStatus.Active` and/or `!vmgr.isVaultExcluded(vault)`.

### [HIGH] H-02 — Resting redeem escrow is stranded after `migrateToken` (stock split)

- **Contract:** OwnMarket.sol
- **Function:** `placeOrder`/`cancelOrder`/`expireOrder`/`fillOrder`/`forceExecuteOrder` via `_activeToken`
- **Line:** 128 (escrow) vs 215/239/390/400 (settle)

**Description:** A redeem order escrows the eToken resolved at placement (line 128). Every later path re-resolves `_activeToken(asset)`. `AssetRegistry.migrateToken` (AssetRegistry.sol:74-83) swaps `activeToken`. After a migration, the contract holds the *old* token but tries to transfer/burn the *new* one → revert or wrong-asset burn. Escrowed legacy eTokens become unrecoverable.

**Impact:** Permanent loss of user escrow on any redeem order open across a stock-split migration; `cancelOrder`/`expireOrder` revert (funds locked forever).

**Recommendation:** Snapshot the escrowed token address in the `Order` struct at `placeOrder` and use it for all returns/burns, instead of re-resolving.

### [HIGH] H-03 — Withdrawal utilization gate combines a stale cached mark with live `totalAssets`

- **Contract:** VaultManager.sol / OwnVault.sol
- **Function:** `withdrawalBreachesUtil()`
- **Line:** VaultManager.sol:466-474; OwnVault.sol:359

**Description:** `withdrawalBreachesUtil` nets against `_globalCollateralUSD` (sum of cached marks). After `releaseCollateralForBadDebt` (OwnVault.sol:544) shrinks real `totalAssets` with **no** mark refresh (the code comment admits "catches up on the next keeper price pull"), `_globalCollateralUSD` is overstated, so the solvency gate reads utilization lower than reality and lets LPs exit ahead of socialized losses.

**Impact:** Informed LPs race out at a stale-favorable utilization snapshot; remaining LPs absorb a larger share; protocol can be pushed past its true solvency cap while the gate still passes.

**Recommendation:** Refresh `pullCollateralPrice(vault)` at the start of `fulfillWithdrawal`, and decrement `_collateralMark`/`_globalCollateralUSD` synchronously inside `releaseCollateralForBadDebt`. Never combine a cached mark with live `totalAssets` in a solvency-blocking check.

### [MEDIUM] M-01 — Pyth confidence interval ignored
PythOracleVerifier `_normalizePythPrice` uses only `price`/`expo`, never `conf`. Wide-uncertainty prices are consumed as exact. **Fix:** reject if `conf·BPS/price > maxConfBps`.

### [MEDIUM] M-02 — In-house push path applies no bounds when an asset's oracle config is unset
OracleVerifier.sol:70,79 — staleness and deviation are gated on `>0` config values; an asset never configured via `setAssetOracleConfig` accepts a single unbounded price. **Fix:** require a configured non-zero `maxStaleness`/`maxDeviation` before accepting prices.

### [MEDIUM] M-03 — `setRateParams` / interest params unvalidated → accrual DoS
BorrowManager.sol:528 + `InterestRateModel.premium` reverts if `optimalUtilBps==0 || >=BPS`. Bad params (or an unvalidated constructor `rateParams_`) make `_accrue` revert, which bricks **repay and liquidate** too (they must stay open). **Fix:** validate params on set and in constructor.

### [MEDIUM] M-04 — Withdrawal queue is not FIFO despite the spec
OwnVault.sol:339 — `fulfillWithdrawal` enforces only wait period + util gate; no ordering. Near the cap, limited capacity is allocated by gas-race/caller choice, letting later LPs jump ahead. **Fix:** enforce oldest-pending-first, or drop the FIFO claim from the spec.

### [INFO — BY DESIGN] C-01* — Force-execute asset proof is an in-window price, not a fresh one

- **Contract:** OwnMarket.sol (`forceExecuteOrder` / `_verifyAssetPrice`)
- **Line:** OwnMarket.sol:29 (window check), `_verifyAssetPrice` (no `priceMaxAge` — intentional)

**Re-audit observation:** the asset proof in `forceExecuteOrder` is gated only by
`assetTs ∈ [order.createdAt, block.timestamp]` with no `priceMaxAge` recency bound, unlike the
collateral leg (`_convertToCollateral`, OwnMarket.sol:463) and `BorrowManager._verifyPrice`
(BorrowManager.sol:799-801). Six agents flagged this as a continuation of C-01 (stale-price replay).

**Resolution — by design (team-confirmed 2026-06-11).** Force-execute is the user's **VM-failure
recourse**: the user placed a redeem order, a VM should have executed it when the asset reached the
user's `limitPrice`, and if no VM did so within `claimThreshold` the user may force-execute by proving
the asset reached that price *at any point in the order's `[createdAt, now]` window*. The price proof
is therefore intentionally window-scoped, not fresh. The original C-01 collateral-theft vector
(settling at an *attacker-submitted* arbitrary price) is closed: the payout is the user's **own,
immutable, placement-time `limitPrice`**, claimable only by `order.user`, only if the asset actually
reached it — there is no unbounded extraction and no third-party theft. The **collateral** leg is
still required fresh (a real release happening now). C-01 stands **Fixed**; this asset-leg behaviour is
the intended design.

**Acknowledged risk (for the record).** By design, LP collateral backstops the redeem guarantee at the
user's committed `limitPrice` whenever the asset reached it during the window — so a price wick that a
VM fails to execute against can trigger an LP payout of up to `(limitPrice − currentPrice) × remaining`
even after the asset has fallen back. This is the intended LP trust model (LPs are the trustless
guarantee that redemptions fill at the agreed price when VMs default), bounded by the user's own
pre-committed limit. No code change. If the team later wants to bound the wick exposure without
removing recourse, the levers are a force-window shorter than the order lifetime, or settling at
`min(limitPrice, fresh price)`; neither is adopted here.

### [HIGH] H-05 — `fulfillWithdrawal` removes collateral without syncing the VaultManager mark

- **Contract:** OwnVault.sol
- **Function:** `fulfillWithdrawal()`
- **Line:** 369 (transfer) vs siblings 541/557

**Description:** `fulfillWithdrawal` transfers vault collateral out (line 369) but never calls
`IVaultManager.onCollateralReleased`, whereas both sibling collateral-exit paths `releaseCollateral`
(541) and `releaseCollateralForBadDebt` (557) do and document why. `_globalCollateralUSD`/`_collateralMark`
stay overstated until a permissionless keeper `pullCollateralPrice`, so subsequent `openExposure` (mint)
and `maxDebtUSD` (borrow) issue exposure/debt against collateral that already left. This is distinct
from H-03 (which fixed the `releaseCollateralForBadDebt` mark sync and the gate's stale-mark/live-totalAssets
mix): here a **normal** withdrawal leaves the mark stale-high and the over-issuance is on the mint/borrow
side, not the withdrawal gate. `rejectDeposit`/`cancelDeposit` correctly skip the sync (untracked pending
assets), isolating `fulfillWithdrawal` as the offender.

**Impact:** Global utilization solvency cap bypassable via serial withdrawals or post-withdrawal
mints/borrows (e.g. 5×100k withdrawals each passing the gate on the un-decremented 1,000,000 mark →
real utilization 140% vs reported 70%).

**Recommendation:** Call `IVaultManager(registry.vaultManager()).onCollateralReleased(assets)` before
the transfer in `fulfillWithdrawal` (skip when `_vaultStatus == Halted`).

### Medium

- **M-06 — `borrow` cap checked pre-accrual.** The hard-cap `totalDebtUSD() + borrowValueUSD > maxDebtUSD()`
  (BorrowManager.sol:25-27) runs before `_accrue()` (line 31); `totalDebtUSD()` reads the stored
  (pre-accrual, pre-floor) index, so accrued interest and the M-05 Aave-debt floor are excluded — a borrow
  that should breach the LTV cap can pass. The M-05 floor widened this from sub-block dust to the full
  rate-spike delta over idle periods. **Fix:** move `_accrue()` above the cap block.
- **M-07 — `borrow` ignores vault status.** `_validateEligibility` checks only the asset
  (`isActiveAsset`/`_assetBorrowDisabled`/`isAssetHalted`/`isTradingPaused`), never
  `IOwnVault(vault).vaultStatus()` — so a vault paused/halted during an incident keeps being borrowed
  against via its Aave credit delegation. (Sibling of H-01, which fixed only the force-execute path.)
  **Fix:** `if (IOwnVault(vault).vaultStatus() != VaultStatus.Active) revert`.
- **M-08 — `_flooredIndex` inflation / multi-manager corruption.** `minIndex = debtToken.balanceOf(vault)
  × PRECISION / _totalScaledDebt` (the M-05 floor) assumes this manager is the sole originator of the
  vault's Aave debt. When `_totalScaledDebt` shrinks to dust while real Aave debt does not, the index
  explodes (e.g. 50e6/1 → ~5e7× jump, ballooning every position); and the documented future second
  borrow manager on the same vault reads the *combined* `balanceOf(vault)`, multiplying its borrowers'
  debt. **Fix:** floor against this manager's own originated debt (tracked), or enforce/skip per a
  one-manager invariant and a dust threshold.
- **M-09 — Sub-unit amount → zero scaled debt.** `actualToScaled(amount, idx)` floors to 0 once
  `idx > PRECISION` and the amount is small, with no `> 0` guard. In `_liquidate` a liquidator pays dust,
  seizes non-zero collateral, but reduces `p.principal` by 0 (borrower loses collateral, debt persists);
  `borrow` can hand out stablecoin / pull collateral while recording `principal = 0` (unrepayable position).
  **Fix:** revert if `scaledRepay == 0`/`scaledDebt == 0` (and `seize == 0`).
- **M-10 — `WstETHRouter` wraps requested not received stETH.** `wstETH.wrap(stETHAmount)`
  (WstETHRouter.sol:102) uses the requested figure, but stETH's share-rounding delivers `stETHAmount − 1..2 wei`,
  so `wrap` reverts — breaking the primary stETH deposit path. `AaveRouter.deposit` uses the correct
  balance-diff pattern (AaveRouter.sol:101-107). **Supersedes §2's "routers verified clean" note for this
  path.** **Fix:** wrap `stETH.balanceOf(this) - balBefore`.

### Low

- **L-08 — Stranded oracle-fee ETH.** `BorrowManager.borrow`/`liquidate`/`absorbBadDebt` forward
  `{value: msg.value}` to `_verifyPrice` (BorrowManager.sol:797); Pyth consumes only the fee and the
  in-house `OracleVerifier.verifyPrice` ignores ETH entirely — the surplus is permanently stranded
  (no refund in BorrowManager, no `receive`/sweep in the verifiers). Same class as the §2 INFO on
  `verifyPriceForSession` dropping `msg.value`. **Fix:** refund `address(this).balance` to `msg.sender`,
  or forward only `verifyFee(priceData)`.
- **L-09 — `settleHaltedPosition` ceil over-seize.** `eTokenToCover` is ceil-rounded so `proceeds` can
  exceed `currentDebt`; the surplus (borrower's seized collateral) is swept to the VM via
  `_repayAaveAndSweep` rather than returned to the borrower. Dust per call, but systematic on every
  non-exact halt settlement. **Fix:** floor `eTokenToCover` when capped, or refund `proceeds − currentDebt`
  to the borrower.
- **L-10 — `settleHaltedPosition` unenforced `paymentToken == stablecoin`.** `redeemHalted` returns
  `proceeds` in the payment token's decimals while `_repayAaveAndSweep`/`actualToScaled` treat it as the
  borrow stablecoin — correct only if both tokens (and decimals) match. The code comments assume it
  ("both USDC in the MVP") but nothing enforces it, and `setPaymentToken` accepts any ≤18-dec token
  (USDS is 18-dec). **Fix:** `require(vaultManager.paymentToken() == stablecoin)`.
- **L-11 — `sweepDividends` destination.** The permissionless `sweepDividends` sends realized dividends
  to `IOwnVault(vault).manager()` (admin-set via `setManager`), a weaker trust boundary than the bad-debt
  path's hard-coded `registry.treasury()`. (Distinct from L-06, which is about validating the *token arg*.)
  **Fix:** route to `registry.treasury()`/`registry.vaultManager()` or enforce the destination equals the VM.
- **L-12 — Fee-on-transfer escrow desync (extends L-02).** `placeOrder` stores `order.amount = amount`
  and settles/returns from it without measuring the received balance (OwnMarket.sol:22,32); a fee-on-transfer
  payment token (USDT's dormant fee switch) leaves the market short → last orders to settle/cancel revert
  or draw down another user's escrow. L-02 flagged only the >18-dec/no-FoT-guard angle for *settlement*;
  this is the *escrow-quantity* desync. **Fix:** store/use `balanceAfter − balanceBefore`, or hard-revert
  if `received != amount`.
- **L-13 — Blocklist-freeze bricks escrow / halt redemptions.** Escrow returns and halt-redeem proceeds
  are pushed synchronously to a single party; a USDC/USDT freeze of that recipient permanently reverts
  `cancelOrder`/`expireOrder` (escrow locked), and a frozen `haltRedeemAddress` bricks
  `redeemHalted`/`settleHaltedPosition` for *every* user of that asset. **Fix:** pull-payment fallback for
  escrow returns; non-freezable, monitored halt address.

### Leads (2026-06-11, unconfirmed — high-signal trails for manual review)

- **No `minAssetsOut` on `fulfillWithdrawal`** — asymmetric vs deposit's `minSharesOut`; `convertToAssets`
  evaluated at fulfillment, so a bad-debt socialization between request and fulfill dilutes a queued LP
  with no opt-out.
- **`shareYield` between `requestDeposit`/`acceptDeposit`** dilutes a pending LP when `minSharesOut == 0`;

- **`_convertToCollateral` floor-div releases 0 collateral** for sub-`1e12`-USD bad-debt slices on a 6-dec

- **eTokens escrowed for resting redeem orders accrue unclaimable dividends to `OwnMarket`** — the market

  orders rest are permanently stranded.
- **`EToken.depositRewards` per-share truncation** — `amount.mulDiv(PRECISION, supply)` floors to 0 for
  large 18-dec supply + low-dec reward token → reward tokens become stuck dust; can quietly zero the

- **`absorbBadDebt` NatSpec vs code mismatch** — doc says the LP-loss collateral "reimburses the caller,"
  code releases it to `registry.treasury()` (admin-only; doc/intent clarity).
